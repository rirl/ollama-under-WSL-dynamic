= Shared Session Manager Design Evolution

An engineering record describing the evolution of the shared session
manager architecture from initial concept through implementation,
including the architectural investigations and design decisions that
led to the current implementation.

:toc:
:sectnums:

== Overview

This document summarizes the evolution of the shared Ollama session manager
through the current state of the `research/shared-session-manager` branch.

The work intentionally followed an iterative engineering process:

* establish a reproducible baseline
* formulate an implementation hypothesis
* preserve intermediate reasoning
* construct architectural models
* perform platform research
* revise the architecture based upon evidence
* implement the revised design

Rather than treating Git purely as source control, the branch was used as an
engineering notebook in which each significant refinement of understanding was
preserved.

== Motivation

The original objective was to make a shared Ollama installation usable across
multiple WSL2 distributions while ensuring that:

* the LLM models reside on a Windows-mounted VHDX
* Ollama executes only while that volume is available
* idle prompt sessions are not mistaken for inactive clients
* multiple WSL distributions cooperate safely
* shutdown is deterministic and idempotent

The initial implementation relied primarily upon TCP socket state
(`ss`, `ESTAB`, etc.) to determine whether Ollama could safely be stopped.

Testing quickly revealed that idle prompt sessions often have no established
TCP connection despite remaining active from the user's perspective.

This observation motivated the introduction of a shared session lease model.

This stage represented the most significant architectural refinement.

The implementation shifted from assumption-driven design to
evidence-driven design.

== Architectural Findings

The research established several important observations.

=== Shared Kernel

WSL2 executes all distributions on a single Linux kernel.

=== PID Namespaces

Each distribution executes within its own PID namespace.

Namespace-local PIDs therefore cannot be assumed to be globally meaningful.

=== NSpid

Linux exposes namespace PID mappings through:

----
/proc/<pid>/status
----

using the `NSpid` field.

This enables correlation between namespace-local PIDs and the root namespace.

=== Session Identity

No documented Windows or WSL mechanism was identified that provides a stable
logical session identifier suitable for lease management.

Consequently, the lease manager introduces its own logical:

----
SESSION_ID
----

independent of Linux PIDs.

PIDs are treated solely as liveness evidence.

== Revised Architecture

The revised implementation introduces a shared control directory on the mounted
LLM volume.

----
.ollama-control/

    sessions/
    lock
    enabled
    draining
----

Lease records now represent idle prompt sessions independently of TCP socket
activity.

Representative lease information includes:

* SESSION_ID
* DISTRO
* LOCAL_PID
* ROOT_NAMESPACE_PID (when available)
* PID_NAMESPACE
* STARTED_AT
* LAST_SEEN
* STATE

Cleanup decisions now consider both:

* network activity
* lease activity

rather than socket state alone.

== Engineering Observations

An important lesson from this effort is that implementation and research were
not separate activities.

Instead the process followed a repeated cycle:

----
Observation
    ↓
Hypothesis
    ↓
Implementation
    ↓
Counterexample
    ↓
Research
    ↓
Architectural Revision
    ↓
Implementation
----

Git history therefore captures not only changes to the software but also the
evolution of the architectural understanding.

== Current Status

The `research/shared-session-manager` branch now contains:

* implementation
* architectural documentation
* UML
* validation guidance
* documentation tooling

The branch should now be considered a research implementation rather than a
completed feature.

Further work should focus on validating the implementation under multiple WSL2
distributions, confirming lease behavior under failure conditions, and deciding
whether the resulting architecture is suitable for promotion into the
`development` branch.

== Conclusion

The resulting architecture is substantially stronger than the initial proposal.

Most importantly, architectural decisions are now grounded in documented Linux
kernel behavior and experimentally validated WSL2 characteristics rather than
assumptions.

This research process also established a repeatable pattern for future KIOS
work:

* preserve the baseline
* document the hypothesis
* prototype the implementation
* perform targeted platform research
* revise the architecture based on evidence
* promote only the validated design into the canonical implementation
